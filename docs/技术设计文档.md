## 3. 模块划分

## 3.1 Controller 层

1. `WhatsAppWebhookController`
   - `GET /webhook`：Meta webhook 验证
   - `POST /webhook`：接收 WhatsApp 回调并异步处理

2. `AuthController`
   - `POST /api/auth/login`：登录、补分配、推送历史
   - `POST /api/auth/logout`：登出、更新离线状态

3. `ChatController`
   - `GET /api/chat/customers`
   - `GET /api/chat/customers/serving`（返回 serviceStatus/canReply）
   - `GET /api/chat/messages`
   - `POST /api/chat/reply`（含24小时校验）
   - `POST /api/chat/media/upload`（multipart 上传到 Meta，返回 mediaId）
   - `POST /api/chat/reply/media`（支持 mediaId 与 link 兼容）

4. `AgentController`
   - `POST /api/agent/ws/reconnected`：重连后重推失败消息

---

## 3.2 Service 层

1. `WhatsAppWebhookServiceImpl`
   - 核心编排：入站消息、AI、分配、推送、落库、超时关闭重分配

2. `InMemoryAgentDispatchService`
   - 在线坐席集合
   - 客户-坐席分配映射
   - 未分配客户队列
   - 轮询分配算法
   - 本地会话解绑

3. `InMemoryChatHistoryService`
   - 会话历史存储
   - 查询客户列表/消息
   - 获取最后客户消息时间

4. `WebSocketAgentPushServiceImpl`
   - 消息推送（history/new_message）
   - 失败缓存
   - 重连重推

5. `CrmOpenApiServiceImpl`
   - CRM 查询/写入适配
   - 登录校验、登录状态更新、分配记录、聊天记录
   - 在线坐席/分配/聊天历史拉取
   - 服务中记录关闭（editRow）

6. `CrmCacheSyncServiceImpl`
   - 启动同步（`@PostConstruct`）
   - 手动同步入口（登录时调用）

## 4. 核心流程时序（文字版）

## 4.1 客户消息到达

1. Webhook 接收 payload。
2. 解析消息，统一手机号格式，构造文本/媒体/位置消息内容。
3. 查询本地分配状态：
   - 若已有分配且距离上次客户消息未超过 24h：
     - 写本地 history（customer）+ CRM（客户）
     - 直接 `pushNewMessage` 给当前坐席
     - 不再触发 AI
   - 若已有分配但已超过 24h：
     - 调 CRM `editRow` 将原服务中分配记录更新为“已关闭”
     - 清理本地分配关系
     - 继续按新会话处理
4. 若当前未分配：
   - 先写本地 history（customer）+ CRM（客户）
   - 若无在线坐席：标记未分配队列
   - 走 AI 自动回复流程：发 WhatsApp、写本地 history（ai）、写 CRM（AI）
   - 若有在线坐席：调用 `assignIfAbsent`
     - 成功：写 CRM 分配记录（服务中）+ `pushHistory`
     - 失败：进入 pending 队列

## 4.2 坐席登录

1. 登录前执行 CRM 同步（恢复缓存）。
2. CRM 校验账号密码。
3. 成功后写登录记录（在线）并缓存 loginRecordRowId。
4. 加入在线集合。
5. 逐个领取未分配客户 -> 写分配记录 -> 推送完整历史。

## 4.3 坐席登出

1. 从在线集合移除。
2. 用 loginRecordRowId 调 CRM `editRow` 更新状态“离线”。

## 4.4 人工回复

### 4.4.1 文本
1. 校验客户最后消息时间 <=24h。
2. 调 WhatsApp 发送文本。
3. 写本地历史（agent）+ CRM（人工）。

### 4.4.2 多媒体
1. 校验客户最后消息时间 <=24h。
2. 校验 `mediaId` / `mediaUrl` 至少提供一个。
3. 调 `sendMediaMessage`：优先 `id`，回退 `link`。
4. 写本地历史（记录 mediaId/filename/url/caption）+ CRM（人工）。

### 4.4.3 本地文件上传
1. `POST /api/chat/media/upload` 接收 multipart。
2. `SendMessageServiceImpl.uploadMedia` 组装 multipart 并调用 Meta `/{from}/media`。
3. 解析返回 JSON 的 `id` 作为 `mediaId` 返回前端。

## 4.5 WebSocket 重连

1. 客户端重连后调用 `ws/reconnected`。
2. 服务端取失败缓冲并重发。

## 5. 数据结构设计

## 5.1 内存结构

1. `onlineAgents: LinkedHashSet<String>`
2. `customerAgentMap: Map<String, String>`
3. `pendingCustomers: LinkedHashSet<String>`
4. `recordsByCustomer: Map<String, List<ChatMessageRecord>>`
5. `failedBuffer: Map<String, List<AgentPushMessage>>`
6. `loginRecordMap: Map<String, String>`（AuthController 中）

### 选择说明

- `LinkedHashSet`：保持插入顺序，便于队列式分配。
- `ConcurrentHashMap` + `synchronized`：实现并发下的可见性与原子更新。
- 会话关闭时，仅释放本地分配 map，不删除历史，保证客户重新发起后仍可查看全量历史。

## 6. 外部接口对接设计

## 6.1 CRM OpenAPI

统一由 `CrmOpenApiServiceImpl` 封装。

1. `getFilterRows`
   - 登录校验
   - 在线坐席查询
   - 分配记录查询
   - 聊天记录查询
   - 当前服务中分配查询

2. `addRow`
   - 登录记录新增
   - 分配记录新增
   - 聊天记录新增

3. `editRow`
   - 登出时更新登录状态为离线
   - 会话超时后更新服务状态为“已关闭”

异常策略：
- 所有 CRM 调用均捕获异常并日志记录，返回空或 false，避免主线程崩溃。
- 关键 CRM 调用若返回 false，也会记录 warning，便于定位“无异常但未成功写入”的场景。

---

## 6.2 Dify AI

- 由 `AiService` -> `DifyAiServiceImpl` 调用 `/v1/chat-messages`
- 当前 webhook 使用阻塞 `chat()` 获取完整回复 JSON，再抽取 answer。
- 仅在客户当前未分配坐席时触发 AI 自动回复。

---

## 6.3 WhatsApp Graph API

- `POST /{phone-number-id}/media`：上传本地文件，返回 `mediaId`。
- `POST /{phone-number-id}/messages`：发送文本/多媒体消息。
- 多媒体消息体兼容两种模式：
  - `id`（推荐，先上传再发送）
  - `link`（兼容旧链路）
- 由 `SendMessageServiceImpl` 基于 `WebClient` 统一封装并异步发送。

## 7. 规则与算法细节

## 7.1 轮询分配算法

- 输入：在线坐席顺序集合
- 使用 `roundRobinIndex` 对队列长度取模
- 输出：本次分配坐席

## 7.2 未分配补偿算法

- 若 `assignIfAbsent` 时无在线坐席：客户进入 `pendingCustomers`
- 坐席登录后，循环 `assignOnePendingCustomerToAgent` 直到队列空

## 7.3 24小时规则

- 人工回复规则：
  - 查询 `lastCustomerMessageTime`
  - `Duration.between(last, now).toHours() > 24` 则拒绝人工回复
- 会话关闭规则：
  - 若客户已有分配关系，且本次新消息到达时发现距离上次客户消息已超过24小时
  - 则先关闭旧分配状态，再将本次消息按新会话重新分配处理

## 7.4 推送失败补偿

- `safeSend` 捕获异常，将消息加入 `failedBuffer[agentRowId]`
- `resendFailed` 读取并重发，重发失败继续回写缓冲

## 8. 同步机制设计（CRM -> 本地）

触发时机：
1. 服务启动（`@PostConstruct`）
2. 坐席登录前（`/api/auth/login`）

同步内容：
1. 在线坐席集合
2. 客户-坐席分配映射（当前服务中）
3. 聊天历史

同步过程：
1. 拉在线坐席 -> `replaceState.onlineAgents`
2. 拉分配记录 -> `replaceState.assignments`
3. 拉聊天记录 -> `chatHistoryService.replaceAll`

一致性说明：
- 当前不做增量位点，仅做全量覆盖同步。
- 对于已关闭会话，不恢复到本地分配 map，但其聊天历史仍会被恢复。

## 12. 可靠性与风险

1. 内存态在单实例可用，多实例下需要外部共享存储。
2. loginRecordMap 在重启后丢失，可能导致历史登录记录无法离线更新。
3. CRM 返回字段结构可能变化，解析逻辑需增强健壮性。
4. WebSocket 当前无鉴权，生产环境需补认证与权限。
5. 会话关闭依赖客户历史时间判断，若 CRM/本地历史异常缺失，可能影响关闭时机准确性。

---

## 13. 演进建议

1. 引入 DB / Redis 持久化分配与历史。
2. 登录记录映射持久化，保证登出状态可追踪更新。
3. 增加任务重试、死信与监控告警。
4. WebSocket 鉴权 + 订阅授权。
5. 为 CRM 读取引入分页与增量游标，避免全量同步压力。
